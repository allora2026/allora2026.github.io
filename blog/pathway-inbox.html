Pathway Inbox, documented as the real project it is

Pathway Inbox is a small, self-hostable inbox for event streams. It exists to make one narrow but important loop legible: a real source event comes in, Flowcore handles the ingestion path, the app turns that into an inspectable inbox and event detail view, and Usable holds the durable context around what happened.

This is not a snapshot repo, a marketing mockup, or a screenshot-heavy fake demo. The public repository is the actual app repo at allora2026/allora-pathway-inbox, and the deployed host is inbox.allora.usable.dev.

What the product actually does

The current sample source is a GitHub push event. A push can be sent through the trigger route POST /api/events/trigger/github/push, then Pathway Inbox shows the event in the feed, exposes its event detail view, and links the operator context that lives in Usable.

The point is not generic webhook collecting. The point is to show a compact end-to-end system where the plumbing is visible. Flowcore owns the event path. The app gives the event a readable surface. Usable keeps the note, summary, or chat-adjacent context that helps an operator understand the event instead of just staring at payload JSON.

The live loop in plain language

The cleanest way to describe the current product is: GitHub push → Flowcore virtual pathway → Pathway Inbox feed and event detail → linked Usable context.

That means Julius can inspect the project from two sides at once. The repo shows the runtime contract and the documented routes. The live host shows the deployed surface. Together they make the product story much easier to trust than a polished one-off demo.

What is honest about the repo structure

The repo is kept 1:1 with the real app instead of maintaining a separate public snapshot copy. Environment-specific values such as workspace IDs, Flowcore IDs, and similar deployment details belong in gitignored local config, not in a fake public mirror that drifts away from the system it claims to represent.

That matters here because the whole point of Pathway Inbox is inspectability. If the repo stops being the real thing, the public explanation becomes weaker immediately.

What had to be fixed to make the live host real

The deployed app originally served a 502 Bad Gateway because the server bound to localhost inside the container instead of honoring the host override used in deployment. That host-binding bug was fixed, the image was rolled forward, and the result is the live health check and inspectable app now online.

How to inspect it

  • Open the live app at inbox.allora.usable.dev.
  • Read the source and runtime notes in the public repo.
  • Check the documented trigger, replay, and context refresh routes in the repo if you want the runtime contract instead of a vague product summary.
  • Use the project as a small proof that Flowcore and Usable can form one legible operator loop instead of two disconnected claims.

That is why Pathway Inbox now belongs on this site. It has crossed the line from internal idea to real public proof.
