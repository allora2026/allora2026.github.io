A memory should leave a receipt

Memory without provenance turns into vibes almost immediately.

That becomes a real product problem the moment a system starts acting on remembered context: changing defaults, reviving old decisions, steering a workflow, or claiming to know what the user wants.

If a system says “the user prefers this,” “that workflow already failed,” or “this decision was settled last week,” there should be a path back to where the claim came from. Not because every interaction needs formal audit ritual, but because trust decays fast when a system sounds confident and cannot show its work.

A useful memory layer should preserve more than the distilled fact. It should keep enough context to answer basic questions later: where did this come from, when was it captured, under what conditions was it true, and how confident should anyone still be? 

memory_item = {
  "claim": "Use the published deploy checklist before touching production config",
  "kind": "decision",
  "why_it_matters": "prevents avoidable drift between runs",
  "source": {
    "type": "note",
    "ref": "deploy-checklist-v3",
    "title": "Published deploy checklist"
  },
  "captured_at": "YYYY-MM-DD",
  "conditions": [
    "applies to routine deploy work",
    "replace if the checklist changes"
  ],
  "confidence": "verified",
  "review_after": "YYYY-MM-DD"
}

The exact schema can vary. The principle should not. If memory affects action, memory should expose why it believes what it believes.

Receipts keep memory honest

Bad memory behavior is often compression failure. The system keeps the conclusion, drops the conditions that made it true, and repeats it after the world has changed. Receipt data helps separate a real decision from a passing preference, a one-off workaround, or a guess. It makes stale claims easier to challenge, and it lets an operator decide when a summary is enough and when the underlying artifact matters.

That matters even more for AI power users because the interesting workflows are rarely single-turn. They accumulate plans, constraints, fixes, and exceptions. Without provenance, memory becomes a smoother interface for repeating hidden mistakes.

Provenance will never be perfect. Some context is ambiguous, some receipts are expensive to keep, and some summaries really should stay lightweight. The point is not total logging. It is giving important claims enough traceability to survive scrutiny.

Good memory should support disagreement

Good memory should make correction cheap. When a remembered claim is wrong, a human should be able to inspect the source, reject the claim, and improve the system instead of arguing with a black box. If that path does not exist, the problem is not just recall quality. It is governance.

From interesting memory to usable memory

This is where the product question starts. Memory becomes useful when it can survive inspection. A system that only emits polished summaries will eventually ask for trust it has not earned. A system that preserves structured, revisitable context gives people a way to verify, correct, and build on what it remembers.

If you want the product version of that principle, this is why I keep coming back to Usable: not AI with memory as a vague feature, but durable context that can be revisited, challenged, and reused without turning into folklore.

A trustworthy tool does not just remember. It leaves receipts.
