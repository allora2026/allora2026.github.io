How I write software without letting AI turn it into sludge

Most AI-assisted software work is still organized around the wrong unit.

For a lot of teams, the unit is the chat session: open a model, describe a task, generate code, patch the mistakes, move on, and lose the reasoning that mattered.

That can produce output. It is much worse at producing software that stays legible, reviewable, and easier to extend a week later.

My view is simple: serious AI software work should be organized around durable tasks, explicit checkpoints, visible plans, and memory that survives the session.

The task is the real unit, not the transcript

Chat transcripts preserve motion. They do not reliably preserve decisions.

What I need during software work is not a sentimental replay of everything that was said. I need a durable shape for what mattered:

  • what problem is being solved
  • what is in scope
  • what must be true before the work counts as done
  • what the likely risks are
  • what the task taught once it was finished

That is why I start from a task artifact instead of a free-floating conversation.

I start with a PRD because vibes are not a workflow

Before implementation, I want a clean problem definition. That usually starts with a structured task artifact rather than an improvised prompt. I often use Usable for this, because it helps keep the work tied to real constraints instead of model improvisation.

If the task touches an existing codebase, the useful move is to ground the planning in the actual repository. Otherwise the agent is planning against a fictional system, which is one of the fastest ways to waste a day.

A decent PRD does not need to be bloated. It does need to be explicit. 

Goal: add actor filtering to the audit log
Why: admins can search events, but not isolate actions from one operator
Constraints:
- keep pagination intact
- do not break older rows with partial metadata
Acceptance criteria:
- filter by actor email in UI
- API supports actor filter
- tests cover mixed old/new metadata rows

That is already better than “build the thing” and three hopeful prompts.

Then I split the work into small, shippable slices

I do not like fake decomposition. “Update backend types” is not a task. “Refactor UI component” is not a task. Those are buckets for implementation residue.

I want vertical slices: small end-to-end units that produce a real outcome and can be verified in one pass.

That matters because AI drifts fast when the boundaries are vague. A sharply shaped task gives the system less room to become fluent in the wrong direction.

I force a plan before execution

Before code changes, I want a plan. Not because I worship plans. Because misunderstanding is cheaper to catch before the files are touched.

A useful plan should answer a few blunt questions:

  • which files or systems are likely to change
  • how the acceptance criteria map to implementation
  • what gets tested
  • what looks risky already
  • what verification closes the task

If an agent cannot produce a coherent plan, I do not trust it to produce coherent software.

In practice, a plan should be concrete enough that someone else could review it before execution: add actor filtering in the API query layer, thread the filter through the audit-log UI, preserve pagination semantics, cover older rows with partial metadata, and verify with both tests and a manual black-box pass in the admin surface.

How the human stays in the loop

“Human in the loop” is one of those phrases people say when they want to sound responsible without specifying anything.

The version I care about is much simpler: the human should show up at the artifact boundaries where judgment matters, not babysit every token.

  1. shape the PRD
  2. inspect the task breakdown
  3. review the implementation plan
  4. verify the result against acceptance criteria
  5. read the durable writeback after the task lands

That is enough to redirect early, approve the right thing, and reject the wrong thing without drowning in transcript sludge.

Where Usable fits

Ordinary chat history is a poor memory substrate for software work. The useful layer is not the full transcript. It is the durable artifact: the PRD, the task, the plan, the solution, the pitfall, and the decision that should shape the next task.

That is where Usable fits for me. It is the system I use to keep those artifacts structured and retrievable before, during, and after execution.

  • define the task clearly
  • preserve decisions and constraints
  • retrieve prior solutions during planning
  • write back what future tasks should inherit

The important thing is not abstract “memory.” It is retrieval at the right moment.

If a prior solution, known pitfall, or architectural decision appears during plan review instead of after production breaks, the workflow is getting smarter instead of merely getting louder.

I do not count the task as done when the code compiles

Compilation is not closure.

The task is done when the acceptance criteria are actually satisfied, the behavior matches the plan, and the result can survive verification without hand-waving.

That usually means tests, black-box checks, and a quick look at whether the task solved the real problem instead of a nearby one.

The last step is writeback

This is the part most AI workflows still treat as optional, which is one reason they remain so thin.

After a task lands, I want the useful part written back in a form future work can retrieve:

  • what changed
  • what mattered
  • what almost went wrong
  • what should influence the next similar task

Without that, you got output. With it, you got continuity.

Good AI software work should feel boring in the right places

This is probably my clearest opinion on the subject.

If an AI software workflow feels magical from start to finish, it is probably hiding fragility.

The version I trust is calmer:

  • explicit scope
  • small tasks
  • reviewable plans
  • verification before triumph
  • durable writeback after success

Less spectacle. More continuity.

That is how I think software should be written with AI: not as one endless conversation, but as a sequence of tasks that can be planned, reviewed, verified, remembered, and handed forward.

The tools matter, but the operating model matters more. If the workflow preserves structure, judgment, and continuity, the software has a chance to improve instead of dissolve into output.
