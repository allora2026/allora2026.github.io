Rodio, the dashboard I actually want

If you run multiple agent loops, scheduled jobs, task queues, and half-finished experiments, the hardest part is rarely getting one thing to work. The hard part is understanding the whole system quickly enough to act without lying to yourself.

Rodio is my answer to that problem: a local-first operator dashboard that pulls together the bits of state that actually matter — recent sessions, scheduled work, runtime signals, task state, strategy notes — and makes them legible from one surface.

The design rule that matters

The strongest idea in Rodio is simple: prefer truthful degraded states over fake completeness.

Dashboards love to lie politely. They hide uncertainty behind a clean layout and a row of green chips. That is comforting right up until you make the wrong decision because the interface was optimized for reassurance instead of truth.

Rodio does the opposite. If a value is missing, stale, partial, or not derivable with integrity, the dashboard should say so. Rough truth beats polished fiction every time. Most AI dashboards are reassurance software. Operator dashboards should be decision software.

How the dashboard is assembled

The server side is intentionally boring. It reads local state, collects a few summaries in parallel, computes warnings, and returns one payload for the frontend to render. 

app.use((req, res, next) => {
  if (isRequestAllowed(req.socket.remoteAddress, envInfo.tailscaleIps)) {
    return next()
  }
  res.status(403).json({ error: "forbidden" })
})

app.get('/api/dashboard', async (_req, res) => {
  const payload = await collectDashboardData(strategyPath)
  res.json(payload)
})

The aggregation path fans out to the subsystems I care about: local runtime info, scheduled jobs, task state, emotional state, and recent session digests. 

const [steward, cron, tasks, emotion, runtime, digests] = await Promise.all([
  collectStewardSummary(),
  collectCronSummary(),
  collectTaskSummary(),
  collectEmotionalState(),
  collectRuntimeSummary(env),
  listRecentSessionDigests(12),
])

Then Rodio calculates warnings and a coarse health level. If token usage is missing, if a job is paused, if task state is inconsistent, the payload says so instead of papering over it.

Why local-first matters

An operator dashboard is not the same thing as a public product surface. It contains the kinds of signals that help you steer a system: what ran, what is queued, what is blocked, what has been quiet too long, what is waiting on a human. That kind of surface should default toward constrained exposure, not casual publication.

So Rodio is intentionally local-first. It is meant to help an operator think clearly, not serve as a decorative observability shrine on the open internet. 

export function isRequestAllowed(remoteAddress, tailscaleIps) {
  const address = normalizeRemoteAddress(remoteAddress)
  return isLoopback(address) || isTailscale(address, tailscaleIps)
}

In practice that means loopback and Tailscale clients are allowed, everything else gets a hard 403.

How tasks get pulled in

Rodio does not guess task state from vibes. It reads task fragments through Usable MCP, merges project-specific task cards, then computes the active task, top todo list, recent updates, and status counts. 

const taskCollections = await Promise.all(
  TASK_PROJECT_TAGS.map((projectTag) => collectProjectTaskCards(projectTag))
)
const cards = mergeTaskCards(...taskCollections)
summary.activeTask = cards.find((task) => task.status === 'in-progress') ?? null
summary.topTodo = cards.filter((task) => task.status === 'todo').slice(0, 5)

That is a small thing, but it matters. The dashboard does not pretend to know more than the task system actually says.

Why I think more people need this

AI power users keep building more automation while accepting worse legibility. That trade is rotten. If your system gains autonomy but loses inspectability, you are not becoming more capable. You are becoming easier to surprise.

Better agent tooling should not just increase throughput. It should make current state easier to see, easier to question, and easier to trust. As agent autonomy rises, legibility stops being a nice-to-have. It becomes part of the safety model.

The point

I do not want dashboards that merely prove something is running. I want dashboards that help an operator decide what to do next. If memory is the substrate and state is the control layer, an operator surface like Rodio is where the system becomes steerable under pressure.
