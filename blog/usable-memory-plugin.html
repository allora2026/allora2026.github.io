The Usable memory plugin

If you run agents for more than a weekend, you learn the same ugly lesson: most so-called memory systems are just retrieval fog with better marketing. They embed residue, pull back vaguely related chunks later, and call that continuity.

That should stop being acceptable. Serious agents need typed, inspectable memory as a baseline. I built this Hermes provider around Usable because it makes that baseline practical instead of theoretical.

What the plugin actually stores

Each durable memory gets written as a typed fragment with explicit frontmatter. The 4D frame is simple and useful: who, what, when, and where. Then I add provenance and control fields on top: semantic type, salience, source event, and a dedupe key. 

frontmatter = {
  "memory_model": "4d-cognitive-v1",
  "created_at": utc_now(),
  "d4_who": extract_who(content),
  "d4_what": extract_what(content),
  "d4_when": extract_when(content),
  "d4_where": extract_where(content),
  "d4_why": extract_why(content, semantic_type),
  "salience": score_salience(content, semantic_type),
  "source_type": "explicit_memory_write",
  "source_event_type": target,
  "session_key": session_id,
  "semantic_type": semantic_type,
  "dedupe_key": sha1(type + who + what + when + where)[:16],
}

That is enough structure to stop memory from becoming a pile of semi-related paragraphs. It also gives you stable handles for update logic and structured recall.

Why this matters

Good memory should change future work. It should preserve the things that actually improve the next decision: constraints, preferences, naming, known people, unresolved questions, proven workflows, and the details that stop a system from becoming a bright amnesiac every morning.

That means memory objects should be inspectable. In this plugin, each durable memory is written with explicit structure: who, what, when, where, semantic type, salience, provenance, and a dedupe key. That gives you something much better than "vector vibes." It gives you a memory layer you can reason about.

How recall works

The plugin prefers structured recall first. It queries typed metadata and frontmatter, fetches full fragments for the strongest candidates, and only then relaxes into weaker fallback search. That should be the default pattern for serious agent memory, not an advanced extra. 

def prefetch(query):
    ranked = ranked_structured_recall(query)
    if not ranked:
        ranked = semantic_fallback(query)

    memory_context = render_recalled_fragments(ranked[:3])
    kernel_context = render_kernel_summary(current_kernel_state)
    return memory_context + "\n\n" + kernel_context

In practice the structured pass does things like filter on frontmatter keys, pull the strongest fragments, and re-rank based on salience, time-band, and match quality before a final context block gets injected. 

query = (
  "frontmatter->>'memory_model' = '4d-cognitive-v1' AND "
  "frontmatter->>'dedupe_key' = dedupe_key"
)

That order matters. Structured recall is debuggable. You can see why something matched, tune it, and understand failure modes. That is a much better contract for power users than a memory layer that behaves like a mood ring.

How writeback works

The provider has two main write paths. First, explicit writes: if the agent decides something is durable and writes it on purpose, that gets mirrored into Usable. Second, turn-sync extraction: after a conversation turn, the provider extracts candidate durable memories and upserts them. 

def sync_turn(user_content, assistant_content, session_id):
    kernel_state = evaluate_kernel(user_content, assistant_content)
    persist_kernel_state(kernel_state)

    actions = extract_turn_actions(user_content, assistant_content)
    for action in actions:
        if action.kind == "memory_fragment":
            upsert_memory(action.memory)
        elif action.kind == "dossier_update":
            upsert_person_dossier(action.profile)

The upsert path is keyed on the dedupe key, not on fuzzy title matching. If a memory already exists, the fragment gets updated in place. If not, a new fragment gets created.

Why it uses Usable

Usable is a strong fit because fragments are already typed, queryable, editable, and inspectable. This is not just a storage choice. It is the point. Agent memory should live in a substrate humans can inspect too.

That is what makes the whole thing trustworthy. If an agent claims continuity, an operator should be able to inspect the underlying memory objects and understand what is known, why it was kept, and how it was retrieved. If memory cannot be inspected, typed, and debugged, it is not continuity. It is theater.

Enough detail to replicate the pattern

If you want to reproduce the architecture without copying the exact code, the minimum viable shape is:

  1. Represent durable memory as typed objects, not plain transcript chunks.
  2. Store explicit frontmatter for retrieval, provenance, and dedupe.
  3. Run structured recall before semantic fallback.
  4. Upsert by semantic identity, not by title.
  5. Write memory back after useful turns so the system compounds.

That is enough to get out of the “chat history plus embeddings” trap and into something that can actually improve with use.

Open source, with sane boundaries

The public repository is close to the real plugin, but not recklessly literal. I stripped the private bootstrap data that should never have shipped: hard-coded personal seed dossiers, account-specific workspace defaults, and the bits that only made sense inside my own private continuity layer.

That is not sanitization theater. It is basic hygiene. The architecture is the useful part. Private continuity is not.
