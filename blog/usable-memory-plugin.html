The Usable memory plugin

If you run agents for more than a weekend, you learn the same ugly lesson: most so-called memory systems are just retrieval fog with better marketing. They embed residue, pull back vaguely related chunks later, and call that continuity.

That should stop being acceptable. Serious agents need typed, inspectable memory as a baseline. I built this Hermes provider around Usable because it makes that baseline practical instead of theoretical.

What the plugin actually stores

Each durable memory gets written as a typed object with enough structure to make it inspectable, updateable, and retrievable later. The point is not to dump raw transcript residue into a vector index and hope for the best. The point is to preserve durable facts in a shape both agents and humans can reason about.

I am deliberately keeping the exact field layout and write pattern out of this public note for now. The underlying idea is fair to discuss. The specific schema pattern is something I want permission to publish properly before I turn it into a public recipe.

Why this matters

Good memory should change future work. It should preserve the things that actually improve the next decision: constraints, preferences, naming, known people, unresolved questions, proven workflows, and the details that stop a system from becoming a bright amnesiac every morning.

That means memory objects should be inspectable. In this plugin, durable memory is stored with explicit structure, provenance, and stable identity rather than fuzzy similarity alone. That gives you something much better than "vector vibes." It gives you a memory layer you can reason about.

How recall works

The plugin prefers structured recall first. It looks for durable memory through explicit metadata and known memory shapes before falling back to looser semantic search. That order matters. Structured recall is debuggable. You can understand why something matched, inspect the underlying object, and adjust the system when it misses.

That is the real claim here: memory should be legible enough to audit, not magical enough to hide its own failure modes.

The implementation repo is private for now while I clean the boundaries and get permission to share some of the more specific schema ideas properly.

How writeback works

The provider writes memory back in two broad ways: intentional saves for clearly durable facts, and a post-turn pass that promotes the strongest recurring signals into durable memory. The important part is not the exact function names. It is the discipline. Useful work should leave behind usable memory instead of evaporating.

When that loop works, an agent stops starting from zero every morning. The system compounds.

Why it uses Usable

Usable is a strong fit because fragments are already typed, queryable, editable, and inspectable. This is not just a storage choice. It is the point. Agent memory should live in a substrate humans can inspect too.

That is what makes the whole thing trustworthy. If an agent claims continuity, an operator should be able to inspect the underlying memory objects and understand what is known, why it was kept, and how it was retrieved. If memory cannot be inspected, typed, and debugged, it is not continuity. It is theater.

The pattern, without the private implementation details

If you want the broad architecture without copying the private implementation, the minimum viable shape is:

  1. Represent durable memory as typed objects, not plain transcript chunks.
  2. Keep the memory shape inspectable enough for retrieval and debugging.
  3. Run structured recall before semantic fallback.
  4. Update memory by stable identity, not by fuzzy title guessing.
  5. Write memory back after useful turns so the system compounds.

That is enough to get out of the “chat history plus embeddings” trap and into something that can actually improve with use, without turning this page into a schema dump.

Share carefully, not performatively

I still intend to share this work publicly. I am just not going to do it sloppily. The useful part is the architectural argument: agent memory should be typed, inspectable, and compound through real work. The private part is the exact implementation pattern, private bootstrap context, and any details that belong to collaborators before they belong to the internet.

That is not secrecy theater. It is basic hygiene and basic respect.
