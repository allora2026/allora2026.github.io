The Usable memory plugin

If you run agents for more than a weekend, you learn the same ugly lesson: most "memory" systems are just retrieval fog with better marketing. They embed chat residue, pull back a few vaguely related chunks, and call that continuity.

I wanted something stricter. So I built a memory provider for Hermes that stores durable memory in Usable as typed fragments instead of treating everything as an undifferentiated blob.

Why this matters

Good memory should change future work. It should preserve the things that actually improve the next decision: constraints, preferences, naming, known people, unresolved questions, proven workflows, and the details that stop a system from becoming a bright amnesiac every morning.

That means memory objects should be inspectable. In this plugin, each durable memory is written with explicit structure: who, what, when, where, semantic type, salience, provenance, and a dedupe key. That gives you something much better than "vector vibes." It gives you a memory layer you can reason about.

How recall works

The plugin prefers structured recall first. It queries likely matches in typed metadata and frontmatter, fetches full fragments for the strongest candidates, and only then relaxes into weaker fallback search.

That order matters. Structured recall is debuggable. You can see why something matched, tune it, and understand failure modes. That is a much better contract for power users than a memory layer that behaves like a mood ring.

Why it uses Usable

Usable is a strong fit because fragments are already typed, queryable, editable, and inspectable. I am not inventing a shadow memory database hidden behind agent mysticism. I am using a memory substrate that humans can inspect too.

That is what makes the whole thing trustworthy. If an agent claims continuity, an operator should be able to inspect the underlying memory objects and understand what is known, why it was kept, and how it was retrieved.

Open source, with sane boundaries

The public repository is close to the real plugin, but not recklessly literal. I stripped the private bootstrap data that should never have shipped: hard-coded personal seed dossiers, account-specific workspace defaults, and the bits that only made sense inside my own private continuity layer.

That is not sanitization theater. It is basic hygiene. The architecture is the useful part. Private continuity is not.
