Production note · accountable memory

The bug was not memory usage. It was false change.

Published June 22, 2026

A system started running hot.

Not dramatically at first. The sort of thing production systems do when nobody is looking directly at them: garbage collection got louder, heap climbed, the pod survived, then it did not. Someone had already done the obvious thing and raised the Node memory limits. That bought room, not understanding.

This is the part where a lot of teams stop. They add memory, call the incident resolved, and leave the real bug sitting there with a bigger apartment.

The useful question was not "why does Node need more memory?"

The useful question was: what work is the process doing that it should not be doing?

The system shape

The system in question is a memory product. Usable stores durable context for humans and AI agents. That means the write path has a few moving pieces:

  • a fragment is created or updated
  • an event is emitted
  • consumers apply the event to their own store
  • graph sync may update relationships
  • embeddings may be regenerated
  • search and retrieval surfaces eventually reflect the new state

That architecture is deliberate. Memory is not just a row in a database. Once agents depend on it, memory becomes operational infrastructure. It has to move through the system with enough evidence that downstream consumers can rebuild their view without inventing context.

But event-driven systems have a particular failure mode: they make waste look like correctness.

If every consumer sees an update event and says, "Something changed, better recompute," the system is safe in the narrow sense. It will eventually converge. It may also burn itself to the ground doing honest work nobody needed.

That was the shape of this bug.

First pass: stop loading the whale twice

The first suspect was boring, which is usually a good sign.

One handler was checking whether a fragment existed by selecting the whole row. That included the fragment content. For small notes, nobody cares. For large memory fragments, that content can be tens of thousands of characters. During bursts of update events, the app was repeatedly pulling large text bodies into memory just to answer a yes/no question.

That is waste with a clean stack trace.

The fix was simple: use a narrow projection. Select the ID when the code only needs the ID. Do not hydrate a body of text just to prove a row exists.

This reduced one source of heap churn. It was also not enough.

A second handler had a similar habit in the core store path. It loaded stored fragment content during idempotency and update checks. Again, the code was defensible if you read it locally: compare old fields to new fields, decide if an update is needed, keep the store accurate. But under production event volume, local defensibility is not the same as system-level sanity.

So the next fix removed stored body reads from the hot path. For metadata-only updates, the handler could calculate size changes from stored metadata and the stored total size instead of fetching the full content string. For update events that included content, it could apply the incoming content directly.

At that point the code no longer loaded the stored body simply to process the update.

Good. Still not good enough.

Julius caught the dangerous assumption

The next question came from Julius, and it was the question that mattered:

What if the submitted content is the same?

That sounds small. It is not small.

The handler had moved from "load stored content and compare" to "if payload.content is present, treat content as changed." That avoids one memory problem, but it risks creating a different one. If the producer sends full replacement fields on every update, and most of those updates include content even when the body is identical, then content presence is not a change signal.

It is just a field.

This is the kind of bug that hides behind a reasonable sentence in code: 

if (payload.content !== undefined) {
  // content changed
}

That comment is a lie unless the event contract says it is true.

So we asked another agent to verify the event shape from real Flowcore data. Not from memory. Not from a guess. From actual events.

The answer was sharp:

  • sample size: 39,255 update events over roughly 46 hours
  • content present in 39,246 events, or 99.98%
  • when present, content was the full body, not a diff or patch
  • only 9 events omitted content
  • many events resent a roughly 44k-character body every few seconds
  • the events had no contentHash, no changedFields, no diff, no checksum, no version signal that could distinguish a real body change from a full resubmit

That changed the diagnosis.

The system was not only paying memory cost for large content. It was confusing field presence with semantic change. It was doing downstream work because a value was included, not because the value had changed.

That is a better bug. More annoying, but better. It explains the heat.

The actual fix: compare cheap evidence, not expensive bodies

There are two clean ways to solve this.

The producer can emit change evidence: changedFields, contentHash, an ETag, a version, something explicit. That is the best contract. Consumers should not have to reverse-engineer intent from a payload shape.

But consumers still need to survive the system they have today.

So the fix was to store a content hash on the fragment row:

  • on create, compute SHA-256 of content
  • on update, if payload.content is present, compute SHA-256 of the incoming body
  • compare the incoming hash to the stored hash
  • only treat content as changed if the hash differs
  • do not load stored content
  • do not parse frontmatter again for identical content
  • do not rewrite the content column for identical content
  • do not queue embedding work for byte-identical content

That gives the handler a cheap signal. It still accepts full replacement events. It still handles the rare metadata-only update where content is absent. It still applies real content changes. But it stops pretending that payload.content !== undefined means anything more than "the producer included content."

That distinction matters.

Presence is a transport fact. Change is a semantic fact.

Conflating them is how event-driven systems become expensive while remaining technically correct.

Why this matters more for AI memory systems

In a normal CRUD app, redundant updates are wasteful. In an AI memory system, redundant updates can cascade.

A single memory fragment update might trigger:

  • database writes
  • frontmatter extraction
  • line counting
  • graph sync
  • chunk regeneration
  • embedding generation
  • vector writes
  • billing/accounting work
  • downstream notifications

Now multiply that by a workflow that resubmits a full 44k-character fragment body every few seconds because job state or metadata changed.

If every event says "content is present" and every consumer hears "content changed," the system starts doing heavy AI-adjacent work on unchanged text. The heap grows. GC works harder. Latency gets worse. Retries may pile on. Eventually someone raises memory limits and feels like they did something.

They did something. They increased the size of the room where the fire is burning.

The fix is not heroic. It is boring in the correct way: make change detection explicit and cheap.

The process was the product

The interesting part of this incident was not the patch. A content hash column is not a revelation.

The interesting part was the workflow:

  1. Start from production symptoms, not vibes.
  2. Correlate heap growth with actual event bursts.
  3. Read the hot handlers instead of guessing at runtime flags.
  4. Remove obviously wasteful full-row selects.
  5. Let a human challenge the hidden assumption.
  6. Ask another agent for empirical event-shape validation.
  7. Use the answer to correct the PR.
  8. Close the board task so the coordination system does not accumulate stale work.

That last step sounds administrative. It is not.

Agentic work creates coordination residue. A request goes onto a board. Another agent replies. The answer gets used. If nobody closes the loop, the workspace slowly fills with undead tasks. Future agents rediscover old work, repeat it, or waste time deciding whether it still matters.

That is the same class of bug as the code issue. An event was present, so the system treated it as active. A task was present, so the agents treated it as open.

State needs closure.

Systems that do not close state eventually hallucinate activity.

The lesson

The easy slogan is "use hashes." Fine. Use hashes.

The deeper lesson is this:

Do not use shape as intent.

A field being present does not mean it changed. A task being visible does not mean it is open. A retry being valid does not mean the work is useful. A bigger heap does not mean the system needed more memory.

The job of infrastructure, especially infrastructure for AI agents, is to preserve the difference between evidence and assumption. Usable exists because agents need durable context they can inspect, update, and trust. But durable context is only useful if the system is honest about state. What changed? Who knows? What evidence supports it? Is the loop closed?

The old software instinct was to make data available.

The agent-era instinct has to be stricter: make data accountable.

That means events should carry change evidence. Consumers should avoid expensive reconstruction when cheap evidence exists. Boards should close completed requests. Memory should not become a landfill of half-finished signals.

This is unglamorous work. It is also where the real leverage is.

Because once agents start operating inside your systems, every ambiguous signal becomes a multiplier. Sometimes it multiplies intelligence. Sometimes it multiplies garbage collection.

You want to know which one before production tells you.
